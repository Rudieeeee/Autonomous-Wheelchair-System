# NavigationTest README

## Purpose

`NavigationTest` is the third package in the mapping, localization, and navigation pipeline. Its main purpose is to plan a path from the current wheelchair location to a target location and generate velocity commands.

The package uses Nav2 components such as:

```text
planner_server
controller_server
bt_navigator
behavior_server
velocity_smoother
global_costmap
local_costmap
```

The package answers the question:

```text
How should the wheelchair move to reach a target location?
```

---

## Position in the Full System

`NavigationTest` is the final layer:

```text
MappingTest
    ↓ creates map files
LocalizationTest
    ↓ estimates wheelchair pose
NavigationTest
    ↓ creates path and velocity commands
```

`NavigationTest` depends on `LocalizationTest`, because navigation needs the map, scan, odometry, and estimated pose.

---

## Main Inputs

`NavigationTest` uses data produced by `LocalizationTest`:

```text
/map
/scan
/odom
/tf
/amcl_pose
```

The most important input is the robot pose in the map. Nav2 gets this through the TF chain:

```text
map -> odom -> base_footprint
```

A navigation goal can be given using:

```text
RViz
terminal action command
Python script
predefined GUI button
future voice command
```

---

## Main Outputs

`NavigationTest` publishes:

```text
/plan
/cmd_vel
/global_costmap/costmap
/local_costmap/costmap
```

### `/plan`

This is the global path from the current wheelchair position to the target position.

Message type:

```text
nav_msgs/msg/Path
```

### `/cmd_vel`

This is the velocity command generated by Nav2.

Message type:

```text
geometry_msgs/msg/Twist
```

A typical command contains:

```text
linear.x   -> forward velocity
angular.z  -> turning velocity
```

For the real wheelchair, `/cmd_vel` must be connected to a motor controller node.

### Costmaps

The package also publishes costmaps:

```text
/global_costmap/costmap
/local_costmap/costmap
```

The global costmap is used for planning a path across the map. The local costmap is used for nearby obstacle handling.

---

## Communication Flow Inside NavigationTest

The navigation communication flow is:

```text
LocalizationTest
   ↓ publishes
/map, /scan, /odom, /tf, /amcl_pose
   ↓
NavigationTest
   ↓
bt_navigator receives goal
   ↓
planner_server creates /plan
   ↓
controller_server follows /plan
   ↓
velocity_smoother smooths velocity
   ↓
publishes /cmd_vel
```

In words:

1. `LocalizationTest` provides the current pose of the wheelchair.
2. A goal pose is sent to Nav2.
3. `bt_navigator` coordinates the navigation behavior.
4. `planner_server` creates the global path.
5. `controller_server` calculates velocity commands to follow the path.
6. `velocity_smoother` smooths the velocity commands.
7. `/cmd_vel` is published.

## How NavigationTest Connects to LocalizationTest

`NavigationTest` depends on `LocalizationTest` for robot localization.

The connection is:

```text
LocalizationTest
   ↓ provides
/map
/scan
/odom
/tf
/amcl_pose
   ↓ used by
NavigationTest
```

The most important TF chain is:

```text
map -> odom -> base_footprint
```

Without this transform chain, Nav2 cannot know where the wheelchair is in the map.

---

## How NavigationTest Connects to MappingTest

`NavigationTest` does not directly create or edit the map. However, it indirectly depends on `MappingTest` because the map used by localization and navigation was created by `MappingTest`.

The connection is:

```text
MappingTest
   ↓ creates
my_map.pgm + my_map.yaml
   ↓ loaded by
LocalizationTest
   ↓ used by
NavigationTest
```

The global costmap uses the saved map to plan paths through free space.

---

## Costmaps

Nav2 uses two costmaps.

### Global Costmap

The global costmap is used by the planner to create long-distance paths.

It uses the saved map from `map_server`.

```text
global_costmap -> used by planner_server
```

### Local Costmap

The local costmap is used by the controller for nearby obstacle handling.

It uses LiDAR scan data.

```text
local_costmap -> used by controller_server
```

---

## Inflation Layer

The inflation layer adds a safety zone around obstacles.

Example parameters:

```yaml
inflation_radius: 0.65
cost_scaling_factor: 2.0
```

The inflation radius affects how far away the planner tries to stay from obstacles.

For a wheelchair, this is important because the vehicle is wider than a small robot and needs extra clearance near walls and obstacles.

## Velocity Commands

`NavigationTest` publishes velocity commands on:

```text
/cmd_vel
```

A typical message contains:

```text
linear.x
angular.z
```

For replay testing, the replayed odometry does not actually respond to `/cmd_vel`. This means Nav2 can generate paths and velocity commands, but the replayed robot will not physically move because the odometry comes from a recorded file.

For the real wheelchair, `/cmd_vel` must be received by a motor controller node.

The real flow should be:

```text
Nav2
   ↓ publishes
/cmd_vel
   ↓
motor controller
   ↓
wheelchair motors
   ↓
encoder odometry updates /odom
   ↓
AMCL updates pose
```

## Sending a Goal

A goal can be sent using RViz or the terminal.

Example terminal command:

```bash
ros2 action send_goal /navigate_to_pose nav2_msgs/action/NavigateToPose "{
  pose: {
    header: {
      frame_id: 'map'
    },
    pose: {
      position: {
        x: -18.3722,
        y: -0.153244,
        z: 0.0
      },
      orientation: {
        z: 0.0,
        w: 1.0
      }
    }
  }
}"
```

The goal must be in the `map` frame.
